# Phase 0 Research: Run-Cost Profile

All unknowns were resolved by direct repository inspection on 2026-07-17 (branch `116-profile-run-cost`). No external research required.

## R1 ‚Äî How the orchestrator invokes dbt (invocation granularity)

**Decision**: Instrument at the `DbtRunner.execute_command` boundary; treat each call as one "invocation."

**Rationale**: `DbtRunner` (`planalign_orchestrator/dbt_runner.py:164`) spawns a **subprocess per `execute_command` call** (`_run_subprocess`, line 467), closing DB connections first to avoid lock conflicts. Call sites are spread across `pipeline/year_executor.py` (6 sites, e.g. lines 268/313/327/461/555/701 + `run_models` at 684), `pipeline/event_generation_executor.py` (3 sites), and `pipeline_orchestrator.py` (3 sites) ‚Äî so a simulated year pays dbt startup + project parse **many times**, which is exactly the hypothesized overhead. Wrapping one method captures all of it.

**Alternatives considered**: parsing dbt's `logs/dbt.log` timestamps (fragile, format-versioned); OS-level process accounting (overkill, loses per-invocation attribution).

## R2 ‚Äî Where per-model compute timings come from

**Decision**: Snapshot `dbt/target/run_results.json` immediately after each invocation; sum per-model `execute` timing entries as the "computation" component.

**Rationale**: dbt-core 1.8 writes per-node timing (`compile` and `execute` phases) into `run_results.json` ‚Äî but **overwrites it on every invocation**, so the harness must copy it per invocation (confirmed present at `dbt/target/run_results.json`). The `execute` phase timing is the DuckDB SQL execution time as measured by dbt itself; spec assumption says this is decision-grade.

**Alternatives considered**: DuckDB profiling pragmas per query (requires modifying model SQL ‚Äî violates FR-009); timing at the SQL cursor level (requires replacing the adapter ‚Äî far too invasive for a measurement).

## R3 ‚Äî Census inputs for the three sizes

**Decision**: tiny = `tests/fixtures/invariant_census.csv` (150 employees, converted to parquet exactly as `tests/fixtures/invariant_simulation.py` does); dev = `data/census_preprocessed.parquet` (7,505 employees, verified by row count); large = NEW synthetic ~60K parquet generated by `scripts/perf_profile/make_large_census.py` scaling the dev census ~8√ó with perturbed unique `employee_id`s and jittered demographics.

**Rationale**: staging reads the census via `read_parquet('{{ var("census_parquet_path") }}')` (`dbt/models/staging/stg_census_data.sql:50`), so any parquet path works with zero product changes. **Caveat discovered**: `data/census_preprocessed_5k.parquet` and `_7k.parquet` are stale copies ‚Äî all three data/ parquets contain the identical 7,505 rows, so the large census genuinely must be generated, and the `_5k`/`_7k` files must NOT be trusted as size variants. ~60K (8√ó) sits near the constitution's 100K-class requirement and matches the clarified "client-representative" intent.

**Alternatives considered**: using a real client census (not available in-repo; PII concerns ‚Äî the spec's edge case already covers this fallback direction in reverse); 100K exactly (8√ó duplication is simpler to make demographically sane; report notes directional transfer).

## R4 ‚Äî Isolation mechanism

**Decision**: Per-repetition fresh DuckDB under `var/perf_profile/db/`, wired via `create_orchestrator(config, db_path=‚Ä¶, threads=1)` + `DATABASE_PATH` env (belt-and-braces), mirroring `tests/fixtures/invariant_simulation.py` (env set/restore at lines 85‚Äì93, orchestrator at 102‚Äì108).

**Rationale**: this is the proven Feature 113 pattern; `get_database_path()` honors `DATABASE_PATH`, and dbt `profiles.yml` follows it too. SC-007 (shared DB byte-identical) is automated via SHA-256 before/after the campaign.

**Alternatives considered**: `planalign simulate --database` CLI (works, but the Python API gives the harness direct access to wrap the orchestrator's `DbtRunner` instance ‚Äî required for M2).

## R5 ‚Äî Probe stage and compiled-SQL execution

**Decision**: Probe EVENT_GENERATION (stage enum `WorkflowStage.EVENT_GENERATION`, `pipeline/workflow.py:44`, wired at 169) for year 2025 on the dev census, executing compiled SQL from `dbt/target/compiled/` in manifest dependency order via the `duckdb` Python client, against a copy of the isolated DB captured post-FOUNDATION.

**Rationale**: EVENT_GENERATION is the roadmap issue's suggested representative stage, has a bounded model set (`dbt/models/intermediate/events/` ‚Äî 12 models), and depends only on FOUNDATION state, making the "pause and copy" point clean. Compiling once with the year's vars resolves Jinja (including `is_incremental()`) before direct execution; incremental pre-hook DELETEs must be replayed manually for the probe's target tables ‚Äî the probe script handles the delete+insert contract explicitly. Any result divergence is itself a headline finding (spec edge case).

**Alternatives considered**: probing FOUNDATION (touches seed loading ‚Äî less representative of steady-state cost); probing a full year (bigger win if it works, but equivalence-checking every table exceeds the 1‚Äì2 day budget; extrapolation from one stage + M2 decomposition is sufficient per FR-005).

## R6 ‚Äî Where harness code and outputs live

**Decision**: Scripts in `scripts/perf_profile/` (new subpackage of the existing `scripts/` convention, alongside `create_census.py` / `generate_invariant_census.py`); raw samples + throwaway DBs in git-ignored `var/perf_profile/`; committed report at `docs/perf/run_cost_profile.md`.

**Rationale**: matches the repo's established conventions (`scripts/` for analysis tooling, `var/` for runtime outputs per CLAUDE.md ¬ß5, `docs/` for committed documents). Keeps product packages untouched (FR-009).

**Alternatives considered**: `tests/` (wrong ‚Äî these aren't tests and must not run in CI); a notebook (violates SC-006 regeneration-without-manual-editing).

## R7 ‚Äî Repetition and warm/cold policy

**Decision**: ‚â•3 timed repetitions per size for headline stats (large size may drop to 2, labeled per spec edge case), plus one untimed-for-headline first run per size labeled `warm=false`. Spread reported as min/median/max.

**Rationale**: FR-002/SC-003 set the floor; the warm/cold split implements the spec's cache edge case. Min/median/max is honest at n=3 (std is theater at that n).

**Alternatives considered**: more repetitions (better CIs but blows the 1‚Äì2 day budget at ~60K-employee scale); single runs (explicitly banned by SC-003).
